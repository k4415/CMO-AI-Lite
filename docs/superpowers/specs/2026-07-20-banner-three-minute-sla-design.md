# バナー10件・3分完了SLA 設計

## 1. 目的

広告運用者が10件のバナー制作を同時依頼したとき、コピー品質、テンプレ構造、選択素材、追加指示、案ごとの違いを維持しながら、画像完成までの待ち時間を短縮する。

成功条件は次の2段階とする。

- 理想条件: 10件を同時依頼したクリック時点から、10件すべてが180秒以内に画像完成する。
- 最低条件: キュー待ちを除き、各バナーの処理開始から画像完成までが180秒以内である。

速度だけを満たして品質を落とした結果は成功に含めない。10件すべてが画像完成し、後述する品質ゲートを通過した場合だけSLA達成と判定する。

## 2. 診断根拠

直近の実行では、再試行がない1回生成でも次を観測した。

- copyplan: 36〜79秒
- Stage 2のOpenAIプロンプト設計: 132〜476秒
- `gpt-image-2 / medium`: 227〜469秒
- 合計: 約9〜15分

Stage 2と画像生成がそれぞれ単独で180秒を超えている。したがって、Stage 2の入力圧縮や不要再試行の削減だけでは目標を達成できない。

## 3. 採用方針

### 3.1 3ノード構成を維持する

正規の処理順は変更しない。

```text
copyplan
  → prompt
  → image
```

ただし、閉じたテンプレ構造を持つバナーの`prompt`ノードでは、OpenAIテキストモデルを呼ばない。確定済み`copyBrief`、テンプレ構造、選択素材、追加指示、表現レギュレーションから、コードで決定論的に`promptJson`を組み立てる。

### 3.2 コピーはAIで一括生成する

`copyplan`は現在のStage Aを維持し、同一グループの最大10案を1回のAI呼び出しで生成する。

入力:

- 選択WHO-WHAT
- ApprovedClaimSnapshot
- テンプレのテキストスロットと文字数制約
- 表現レギュレーション
- 追加指示原文
- baselineとvariationの候補番号

出力:

- 共通の勝ち筋仮説とビジュアル意図
- 案ごとの`appealAxis`
- 案ごとの`variationDirection`
- 案ごとの`whyItStops`
- テンプレの各`slotId`へ対応する確定`slotTexts`

AIが生成したコピーは、スロット不足、文字数、禁止表現、同一コピー重複、戦略参照、追加指示を自動検証する。品質ゲートに不合格の案を警告だけで画像生成へ流さない。SLA検証では10案すべてが初回copyplanで合格することを必須とする。

Stage 2以降は、確定コピーの変更、省略、要約、追記を禁止する。

### 3.3 閉じたテンプレ用の決定論的prompt compiler

`templateZones`が存在するテンプレでは、`promptJson`を純粋関数で生成する。

入力:

- `banner`
- 商品識別情報
- 選択WHO-WHAT
- `copyBrief`
- `copySlotPlan`
- `creativeHypothesis`
- `approvedClaimSnapshot`
- `templateZones` / `templateGlobalDesign`
- `instructionPolicy`
- 表現レギュレーション
- 選択素材ポリシー

出力:

- 現行保存契約と互換性のある`promptJson`
- `promptText`
- `bannerGenerationContract`
- `colorDecision`
- `promptGenerationAudit`

組み立て規則:

1. zone数、element数、slotId、type、role、位置、サイズ、effectはテンプレを正とする。
2. text要素は同じ`slotId`の`copyBrief.slotTexts`を一字も変えずに配置する。
3. image要素の説明は、テンプレの役割を維持しつつ、`creativeHypothesis.visualIntent`、`copyBrief.appealAxis`、`variationDirection`、`targetMoment`を使って商品・戦略向けに差し替える。
4. shape要素の構造的内容はテンプレから変更しない。
5. 配色・書体・余白は`templateGlobalDesign`を優先し、追加指示による明示変更だけを許可する。
6. ユーザー選択ロゴ・商品画像・その他画像は、テンプレ構造より優先する唯一の例外として必ず含める。
7. 選択されていない画像、ロゴ風文字、装飾、ゾーン、テキスト枠を追加しない。
8. `promptText`は確定`promptJson`から生成し、別AIに再解釈させない。

現行の全配布テンプレに対して、決定論的コンパイル後もzone数、element数、type別件数、slotIdが一致する不変条件テストを行う。

`templateZones`がない旧データ・テンプレ未指定生成では、後方互換のため既存Stage 2 AIを残す。ただし、このfallback経路は今回の3分SLA対象外とし、SLAテストには閉じたテンプレだけを使用する。

### 3.4 画像生成品質プロファイル

通常の新規バナー生成と自動回復に使う`gpt-image-2`の品質を、サーバー設定から`low`または`medium`で選べるようにする。UIは追加・変更しない。

- 検証候補: `low`
- ロールバック値: `medium`
- モデル: `gpt-image-2`固定
- 画像内文字・ロゴの後処理合成: 禁止

`low`は速度だけで採用しない。実画像10件の全件が品質ゲートを通過し、SLAを満たしたラウンドが確認できた場合だけ既定値として採用する。品質に1件でも重大な不合格があれば`medium`へ戻し、現行モデル制約下ではSLA未達と判定する。

画像生成監査へ、実際に送信した`quality`、リクエスト開始・終了、所要時間、request ID、入力manifest hash、試行理由を保存する。APIキー、プロンプト本文、絶対パスは保存しない。

### 3.5 10並列

既存のprompt worker、image worker、ファイルセマフォを利用し、上限10を維持する。新しいUI操作は追加しない。

10件一括依頼時は次の順で動作する。

1. 同一グループのcopyplanを1回実行する。
2. 10件の決定論的prompt compilerを実行する。
3. prompt完了順に止めず、対象10件を画像キューへ投入する。
4. image workerとプロセス間セマフォの両方で最大10件を同時開始する。

テストではモックAPIを使い、同時実行数が実測10に達し、11件目が開始されないことを確認する。実画像テストでも10件の画像`startedAt`が同一バッチ内で重なることを確認する。

## 4. SLA計測契約

時計は可能な限り単調増加時計を使い、保存データのISO時刻と外部テストハーネスの双方で照合する。

### 4.1 理想条件

```text
batchMakespanMs = 最後の画像completedAt - 一括APIリクエスト送信時刻
```

判定:

- 対象件数が10件である。
- 10件すべてが`completed`または`completed_with_warnings`へ到達する。
- `batchMakespanMs <= 180000`である。
- ただし品質警告が実画像の重大不一致を示す場合は不合格とする。

### 4.2 最低条件

```text
perBannerProcessingMs = image completedAt - そのバナーの最初の実処理startedAt
```

キュー受付時刻からworker取得までの待機時間は除く。copyplanを共有する場合、そのグループのcopyplan開始を各バナーの最初の実処理開始とする。

判定:

- 10件すべてで`perBannerProcessingMs <= 180000`である。
- 処理失敗、未完了、品質不合格を時間内完了として扱わない。

### 4.3 診断用内訳

合否とは別に、次を記録する。

- copyplan所要時間とAI/HTTP試行回数
- prompt compiler所要時間
- imageキュー待ち時間
- image API所要時間と品質値
- OCR・ロゴ検証所要時間
- バッチ最大同時実行数
- バッチ全体makespan

## 5. 品質ゲート

### 5.1 機械判定

10件すべてで次を確認する。

- copyBriefの全必須slotが存在する。
- 確定コピーとpromptJsonのslot textが完全一致する。
- テンプレの閉じた構造が不変である。
- 選択素材がすべて入力manifestへ含まれる。
- 未選択素材が入力manifestへ含まれない。
- 追加指示が`instructionPolicy`と最終プロンプトへ残る。
- baselineとvariantのコピー署名が重複しない。
- `variationDirection`または視覚指示が案ごとに識別できる。
- コピーOCRと正式ロゴ検証を実行する。

### 5.2 人間による実画像レビュー

各画像を次の観点でPASS / WARN / FAIL判定する。

- 日本語の文章と画像内文字に違和感がない。
- ビジュアル、人物、商品、空間、視線誘導に違和感がない。
- 選択ロゴの正式ワードマーク、比率、配置に重大な違和感がない。
- 選択商品画像が別商品化・欠落・不自然な重複をしていない。
- 追加指示が正しく反映されている。
- テンプレにない未選択要素が追加されていない。
- WHO-WHATのターゲットが反応する具体的なフックと便益がある。
- 10案が同一コピー・同一見せ方の焼き直しになっていない。

文字化け、意味が変わる誤字、選択素材の欠落・別物化、追加指示の未反映、戦略逸脱はFAILとする。OCRの読み取り不能だけで目視上正しい場合はWARNにできるが、理由を記録する。

## 6. テストと修正ループ

有料の実画像生成は、ユーザー承認済みの最大30件に制限する。

ここでいう1件は、`gpt-image-2`へ送る新規生成または自動回復の画像APIリクエスト1回を指す。自動回復が発生した場合も同じ30件の予算から差し引き、1ラウンドの途中でも累計30件へ達した時点で新しい有料リクエストを停止する。テキストAIとモックテストはこの件数に含めない。

```text
Round 1: 10件
  → 速度・品質レビュー
  → 原因分類・修正
Round 2: 10件（必要な場合だけ）
  → 回帰レビュー
  → 原因分類・修正
Round 3: 10件（必要な場合だけ）
  → 最終確認
```

各ラウンドは3種類以上のテンプレを含める。

- テキスト・shape中心、画像枠なし
- 画像枠あり、ロゴ選択あり
- 画像枠あり、ロゴと商品画像の選択あり

さらに、追加指示あり、複数variation、異なるWHO-WHATを10件内へ配分する。

実画像ファイル、検証用案件、個別案件JSONはGitへ追加しない。Gitへ残すのは匿名化した集計結果と再現可能なテストコードだけとする。

## 7. TDD実装単位

実装は各単位でREDを確認してからGREENにする。

1. 決定論的compilerがOpenAIテキストAPIを呼ばず、現行promptJson契約を生成するテスト。
2. 全配布テンプレの構造不変条件とslot text完全一致テスト。
3. WHO-WHAT、creativeHypothesis、variation、追加指示、選択素材が最終プロンプトへ残るテスト。
4. テンプレなしfallbackだけが既存Stage 2 AIを呼ぶテスト。
5. `low` / `medium`の画像品質値がgenerationとasset付きeditの両方へ渡るテスト。
6. 画像監査へqualityと所要時間が保存されるテスト。
7. worker上限10と11件目待機の負荷テスト。
8. 一括依頼からprompt完了後に10件が画像キューへ投入される統合テスト。
9. SLA計測関数が理想条件・最低条件・失敗条件を正しく分類するテスト。
10. 既存のコピー、テンプレ構造、ロゴ、選択素材、OCR、再生成テストの全回帰。

## 9. 既存仕様との優先関係

次を維持する。

- 選択素材はテンプレ構造より優先する唯一の例外である。
- 選択素材以外は閉じたテンプレ構造へ追加しない。
- 確定copyBriefをpromptノードで書き換えない。
- 事実DBをバナー生成へ渡さない。
- `copyplan → prompt → image`の3ノードを維持する。
- `gpt-image-2`を使用し、文字・ロゴを後処理合成しない。
- UIへ新機能を追加しない。

本書は、閉じたテンプレのStage 2をAI設計1回とする規則と、`quality: medium`固定規則だけを更新する。`docs/banner-generation-latency-retry-fix-instructions-2026-07-20.md`のその他の品質・監査・再試行規則は維持する。

## 10. 完了条件と停止条件

完了には次のすべてが必要である。

1. 対象テスト、全テスト、構文確認、`git diff --check`がPASSする。
2. 全配布テンプレの決定論的構造不変条件がPASSする。
3. モック負荷テストで最大10並列を確認する。
4. 実画像10件の全件が品質ゲートを通過する。
5. 実画像10件が理想条件を満たす。または、最低条件を満たし理想条件未達のキュー原因を明示できる。
6. 最終セルフレビューが8.5 / 10以上で、P0 / P1の未解決事項がない。

最大30件を実行しても、`gpt-image-2 / low`の単一リクエストが180秒を安定して超える、または`low`が品質ゲートを満たさない場合、速度のために品質を下げ続けない。この場合は次を明示して停止する。

- コード上のボトルネックは解消済みか。
- 外部画像API時間が占める割合。
- 理想条件と最低条件のどちらが未達か。
- 品質不合格の具体例。
- 次に必要な意思決定が画像モデル・プロバイダー変更かどうか。

未達を達成済みとして報告しない。

## 11. 設計セルフレビュー

初回レビュー: 9.2 / 10、実装計画作成可。

- 正しさ: 9.2。実測ボトルネックであるStage 2と画像品質を直接対象にしている。
- 要件適合: 9.4。3ノード、10並列、品質、素材、追加指示、多様性、UI据え置きを網羅している。
- 検証可能性: 9.5。SLA式、10件×最大3ラウンド、品質FAIL条件が明確である。
- 保守性: 8.8。閉じたテンプレだけを純粋compilerへ分離し、旧データfallbackを残す。
- 安全性: 9.1。有料上限、品質不合格時のロールバック、個別案件データ除外を固定している。

セルフレビューで修正した点:

1. 単純にStage 2を削除すると案ごとのビジュアル差が弱くなるため、`creativeHypothesis.visualIntent`と案別`variationDirection`をimage要素・design rationaleへ必ず反映する規則を追加した。
2. `low`を速度だけで既定化しないよう、10件全件の実画像品質ゲートと`medium`ロールバック条件を追加した。
3. キュー待ち除外の開始点が曖昧だったため、共有copyplan開始を各バナーの処理開始と定義した。
4. 10件のうち失敗案を時間内完了へ含めないよう、完了件数と品質判定をSLA式へ追加した。
5. 自動再生成で有料上限を超えないよう、ラウンド単位ではなくAPIによる新規・回復画像の総数を30件に含めることを明確化した。
